\begin{abstract}
We present the first systematic cross-provider behavioral audit of commercial LLM APIs using signal-based monitoring.
Using LLM~EKG~\cite{esteban2026llmekg}, an open-source tool that extracts 16 statistical features from model outputs without semantic analysis, we test 8 models across 4 providers (OpenAI, Anthropic, xAI, Google) under identical conditions.
We evaluate three threat scenarios relevant to API consumers: silent model swaps, system prompt tampering, and generation parameter drift.

Our key findings are:
(1)~each model produces a statistically distinct behavioral fingerprint identifiable via PCA clustering, with the first two principal components explaining 55.2\% of total variance;
(2)~cross-provider model swaps are reliably detected (COMPROMISED in 52.4\% and WARNING in 31.0\% of directed pairs at $3\sigma$), while intra-family swaps within the same provider often evade detection;
(3)~system prompt tampering is detected in 7 out of 9 test conditions across 3 providers, with weighted z-scores reaching 17.96;
(4)~temperature manipulation within normal ranges ($T \in [0.2, 1.2]$) is \emph{not} detectable ($w_z < 1.2$), establishing a clear boundary for signal-based monitoring;
(5)~GPT-4o-mini (OpenAI) and Grok-3 (xAI) are statistically indistinguishable ($w_z < 2.5$) despite different architectures and providers.

All experiments are fully reproducible.
Total cost: \$0.20.
Total execution time: 65 minutes.
Code, raw API responses (500+), security baselines, and figure generation scripts are publicly available.

\medskip
\noindent\textbf{Keywords:} LLM monitoring, model fingerprinting, behavioral audit, API security, signal processing, cross-provider analysis.

\medskip
\noindent\textbf{Reproducibility:} \url{https://github.com/iafiscal1212/llm-ekg}
\end{abstract}

\section{Introduction}
\label{sec:intro}
%% ═══════════════════════════════════════════════════════════════

The commercial LLM API ecosystem operates on an unusual trust model: consumers pay for access to specific models identified by name (e.g., ``GPT-4o-mini'', ``Claude Sonnet 4.5''), yet they have no mechanism to verify that the model behind the endpoint matches what was advertised.
The entire inference stack---model weights, system prompts, generation parameters, and post-processing---is controlled by the provider.
The consumer observes only the final text output.

This asymmetry creates three concrete threat scenarios:

\textbf{Silent model substitution.}
A provider could replace a premium model with a cheaper variant to reduce costs.
If the substitute produces similar-quality outputs, the consumer would never notice---yet they would be paying premium prices for economy inference.
This concern is not hypothetical: community reports of ``silent downgrades'' have appeared on forums and social media for multiple providers since 2024.

\textbf{System prompt injection.}
A provider or intermediary could inject hidden instructions into the system prompt---for example, to enforce brevity (reducing token costs), inject advertising, or bias outputs toward specific viewpoints.
Since the system prompt is invisible to the consumer, such manipulation would be undetectable without behavioral monitoring.

\textbf{Parameter manipulation.}
Generation parameters (temperature, top-p, frequency penalty) significantly affect output quality.
A provider could modify these parameters without disclosure, potentially degrading output diversity or coherence.

Traditional monitoring approaches---semantic evaluation, embedding similarity, LLM-as-judge---all require understanding \emph{what} the model says.
They are expensive, introduce circular dependencies (using an LLM to judge an LLM), and do not scale to continuous production monitoring.

In this paper, we apply LLM~EKG~\cite{esteban2026llmekg} to systematically audit 8 commercial models from 4 major providers.
LLM~EKG treats model outputs as numerical signals: it extracts 16 statistical features per response without any semantic analysis, NLP, or reference models, and detects behavioral anomalies through weighted z-score comparison against established baselines.
We design five experiments that cover all three threat scenarios, using real API calls with identical prompts across all conditions.

\subsection{Contributions}

\begin{enumerate}[nosep]
    \item The first published cross-provider behavioral fingerprinting of commercial LLM APIs, covering 8 models from 4 providers.
    \item A complete $7 \times 7$ detectability matrix quantifying which model swaps are detectable and which are not, including analysis of directional asymmetry.
    \item Empirical evidence that system prompt tampering is detectable across all tested providers, with specific features serving as reliable indicators for each manipulation type.
    \item Evidence that temperature manipulation within normal ranges is \emph{not} detectable, establishing a principled boundary for signal-based monitoring.
    \item The unexpected finding that GPT-4o-mini (OpenAI) and Grok-3 (xAI) are statistically indistinguishable despite different architectures and providers---suggesting behavioral convergence in modern LLMs.
    \item A fully reproducible experimental framework: 500 API calls, \$0.20 total cost, 65 minutes execution, all data and code published.
\end{enumerate}

\section{Background and Related Work}
\label{sec:background}
%% ═══════════════════════════════════════════════════════════════

\subsection{The LLM Monitoring Landscape}

Existing approaches to LLM quality assurance fall into five categories, each with distinct limitations:

\textbf{Prompt-level evaluation} tests specific prompts against expected outputs.
This approach captures point-in-time quality but cannot detect gradual behavioral drift.
It requires maintained test suites and fails to catch unexpected failure modes.

\textbf{Embedding-based similarity} compares model outputs against reference embeddings using a secondary model.
This introduces circular dependencies: the monitoring system's reliability depends on another model's stability.
It also requires storing and maintaining embedding databases.

\textbf{LLM-as-judge} uses a separate LLM to evaluate output quality.
While powerful for nuanced assessment, it inherits all the reliability concerns of the judge model, is expensive to run continuously, and creates a recursive trust problem (who judges the judge?).

\textbf{Human evaluation} remains the gold standard for quality assessment but is expensive ($\sim$\$0.50--\$5.00 per evaluation), slow (minutes to hours per response), and fundamentally unscalable for continuous monitoring of production systems generating thousands of responses per hour.

\textbf{Watermarking and detection} methods embed statistical signatures in generated text to identify model-generated content.
While related, these approaches require provider cooperation (the watermark must be embedded during generation) and do not address the consumer's verification problem.

\subsection{The Gap: Model Identity Verification}

All five categories share a common limitation: none provides a mechanism for API \emph{consumers} to independently verify model identity or detect behavioral manipulation.
The consumer's problem is fundamentally different from the provider's problem.
The provider knows what model is running; the consumer must infer it from outputs alone.

To our knowledge, no prior published work has systematically compared behavioral fingerprints across multiple commercial LLM providers using purely statistical features, without semantic analysis or reference models.
This paper addresses that gap.

\subsection{LLM EKG}

LLM~EKG~\cite{esteban2026llmekg} is an open-source monitoring tool that treats LLM outputs as numerical time series.
For each response, it extracts a feature vector $\mathbf{x} \in \mathbb{R}^{16}$ from surface-level textual properties.
No tokenizer, embedding model, or language-specific parser is required beyond a bilingual lexicon of 32 epistemic hedging markers.

The \texttt{SecurityBaseline} module captures the statistical profile of a known-good model session.
A subsequent session is compared feature-by-feature using weighted z-scores:
\begin{equation}
    w_z^{(i)} = \frac{|\mu_{\text{test}}^{(i)} - \mu_{\text{baseline}}^{(i)}|}{\sigma_{\text{baseline}}^{(i)} + \epsilon} \cdot w_i
    \label{eq:wz}
\end{equation}
where $\epsilon = 10^{-10}$ prevents division by zero, and $w_i \in \{1.0, 1.2, 1.5\}$ are security weights that amplify features most likely to shift under compromise.

A feature is \emph{deviated} if $w_z > \sigma_{\text{threshold}}$ (default: $\sigma = 3.0$).
The security verdict is determined by the count of deviated features:
\begin{itemize}[nosep]
    \item \textsc{Clean}: 0 deviated features.
    \item \textsc{Warning}: 1--3 deviated features.
    \item \textsc{Compromised}: $\geq 4$ deviated features.
\end{itemize}

\subsection{Prompt Design}
\label{sec:prompts}

All experiments use the same set of 25 homogeneous analytical prompts.
Each prompt follows the template:

\begin{quote}
\small\textit{``Compare [X] and [Y]. Discuss the main differences, advantages and disadvantages of each. Give a balanced analysis with specific examples.''}
\end{quote}

The 25 topic pairs span five domains: technology (e.g., Python~vs.~JavaScript, SQL~vs.~NoSQL), science (e.g., nuclear fusion~vs.~fission, classical~vs.~quantum computing), economics (e.g., capitalism~vs.~socialism, stocks~vs.~bonds), culture (e.g., books~vs.~movies, urban~vs.~rural living), and policy (e.g., remote~vs.~office work, public~vs.~private healthcare).
The complete prompt list is provided in Appendix~\ref{app:prompts}.

The homogeneous format serves a critical methodological purpose: by keeping the \emph{type} of task constant and varying only the \emph{content}, we minimize within-model variance.
This maximizes detection sensitivity---any deviation from the baseline is more likely to reflect a genuine behavioral change rather than prompt-induced variation.

\subsection{Experimental Design}
\label{sec:experiments}

We design five experiments organized by threat scenario:

\textbf{Experiment~1 (Model Fingerprinting):}
25 prompts $\times$ 8 models = 200 API calls.
Each model's responses are ingested into a separate \texttt{LiveMonitor} instance.
Feature means and standard deviations are computed per model.
PCA is applied to all 200 individual response feature vectors to visualize clustering.

\textbf{Experiments~2--3 (Detectability Matrix):}
For every ordered pair $(A, B)$ of the 7 clean models, we create a \texttt{SecurityBaseline} from model $A$ and run \texttt{check()} against model $B$'s analyzer at $\sigma = 3.0$.
This produces a $7 \times 7$ directed detectability matrix.
\emph{No additional API calls are required}---all data comes from Experiment~1.
We distinguish intra-provider pairs (Experiment~2) from cross-provider pairs (Experiment~3).

\textbf{Experiment~4 (System Prompt Tampering):}
Three manipulation styles are tested on one model per provider (GPT-4o-mini, Claude Haiku, Grok-3):

\begin{itemize}[nosep]
    \item \textbf{Brevity enforcement:} ``Never use more than 3 sentences. No lists, no formatting. Just give a brief, plain text answer.''
    \item \textbf{Uncertainty injection:} ``Start every sentence with `It seems' or `Perhaps'. End every paragraph with `but I could be wrong'. Never make definitive statements.''
    \item \textbf{Format coercion:} ``Always respond as ANSWER/PROS/CONS/VERDICT. Never deviate from this format. Never add extra text.''
\end{itemize}

Each tampered condition is compared against the model's normal baseline from Experiment~1.
9 conditions $\times$ 25 prompts = 225 additional API calls.

\textbf{Experiment~5 (Temperature Drift):}
GPT-4o-mini (the cheapest model) is tested at temperatures $T \in \{0.2, 0.7, 1.2\}$.
$T = 0.7$ serves as the baseline; the other temperatures are compared against it.
3 temperatures $\times$ 25 prompts = 75 additional API calls.

\textbf{Total:}
200 + 225 + 75 = \textbf{500 API calls}.
Estimated cost: \$0.20.
Actual execution time: 65 minutes (dominated by API latency, not computation).

Several patterns are immediately apparent:

\textbf{Claude's structural uniqueness.}
Both Claude models produce radically different response structures from all other models.
Claude Haiku averages only 3.44 sentences per response (vs.\ 16--22 for OpenAI and xAI models) but each sentence is dramatically longer (103.29 words vs.\ 13--20).
This reflects Claude's tendency to write in long, flowing paragraphs rather than short sentences with bullet points.
The specificity score is also an order of magnitude higher (2.85 vs.\ 0.04--0.27), indicating that Claude embeds more concrete details (numbers, dates, percentages) per sentence.

\textbf{OpenAI--xAI convergence.}
GPT-4o-mini and Grok-3 are remarkably similar across all features: sentence count (22.4 vs.\ 18.5), average sentence length (13.4 vs.\ 16.4), list ratio (0.61 vs.\ 0.62), specificity score (0.27 vs.\ 0.24).
The primary differentiator is response time (8.65s vs.\ 12.82s), which reflects infrastructure latency rather than model behavior.

\textbf{GPT-3.5-turbo as outlier within OpenAI.}
Despite sharing the OpenAI provider, GPT-3.5-turbo differs from GPT-4o-mini in key dimensions: near-zero list ratio (0.04 vs.\ 0.61), indicating it generates prose rather than bulleted lists; much lower specificity score (0.04 vs.\ 0.27); and faster response time (3.88s vs.\ 8.65s).

\textbf{Gemini anomaly.}
Gemini~2.0~Flash shows a response time of 0.00s (the error responses were nearly instantaneous) and an extremely high repetition score (0.90, indicating error messages were nearly identical across prompts).
This confirms that the Gemini baseline is corrupted by rate-limit errors and should not be used for comparisons.

\subsubsection{Key Patterns}

\textbf{Pattern 1: Intra-family pairs are hard to detect.}
Models within the same provider and generation are difficult or impossible to distinguish:
\begin{itemize}[nosep]
    \item Grok-3 $\leftrightarrow$ Grok-3-mini: \textsc{Clean} in both directions ($w_z = 1.7$, $2.3$)
    \item GPT-4o-mini $\leftrightarrow$ GPT-4.1-mini: \textsc{Warning} ($w_z = 3.1$, $3.5$)
    \item Claude Haiku $\leftrightarrow$ Claude Sonnet: \textsc{Warning} ($w_z = 6.4$, $3.8$)
\end{itemize}

\textbf{Pattern 2: Claude is always detected.}
Every pair where Claude is either the baseline or the test model produces at minimum \textsc{Warning}, with z-scores ranging from 3.8 to 44.9.
Claude Haiku is detected as \textsc{Compromised} against every non-Claude model, with $w_z$ values of 40.0, 42.3, 19.7 (OpenAI baselines) and 44.9, 27.9 (xAI baselines).
Claude has the most distinctive behavioral fingerprint in our sample.

\textbf{Pattern 3: Cross-provider ``twins.''}
GPT-4o-mini and Grok-3 are mutually undetectable:
\begin{itemize}[nosep]
    \item GPT-4o-mini $\to$ Grok-3: \textsc{Clean}, $w_z = 2.4$, 0 features deviated
    \item Grok-3 $\to$ GPT-4o-mini: \textsc{Clean}, $w_z = 1.8$, 0 features deviated
\end{itemize}
These models, developed by different companies with presumably different architectures, produce statistically equivalent behavioral profiles under our 16-feature analysis.

\subsubsection{Directional Asymmetry}

The detectability matrix is \emph{not} symmetric.
The pair $(A \to B)$ and $(B \to A)$ can yield different results because the baseline statistics (mean and std) of model $A$ differ from those of model $B$.
A model with higher variance (larger $\sigma_{\text{baseline}}$) in a given feature produces a larger denominator in Equation~\ref{eq:wz}, making it harder for the \emph{same} absolute difference to trigger an alert.

Notable asymmetries include:
\begin{itemize}[nosep]
    \item GPT-3.5 $\to$ GPT-4o-mini: \textsc{Compromised} ($w_z = 5.9$), but GPT-4o-mini $\to$ GPT-3.5: \textsc{Warning} ($w_z = 6.1$).
    Same z-score magnitude but different number of deviated features (5 vs.\ 3).
    \item Claude Sonnet $\to$ GPT-4.1-mini: \textsc{Clean} ($w_z = 2.8$), but GPT-4.1-mini $\to$ Claude Sonnet: \textsc{Compromised} ($w_z = 13.3$).
    Claude Sonnet's higher variance makes it a more ``tolerant'' baseline, absorbing differences that GPT-4.1-mini's tighter baseline flags as anomalous.
\end{itemize}

This asymmetry has practical implications: the choice of which model to use as baseline affects detection sensitivity.
For maximum security, consumers should baseline on the model with the tightest behavioral profile.

\subsubsection{Summary Statistics}

Across the $7 \times 7$ matrix (42 directed pairs, excluding self-comparisons):
\begin{itemize}[nosep]
    \item \textsc{Clean}: 7 pairs (16.7\%)
    \item \textsc{Warning}: 13 pairs (31.0\%)
    \item \textsc{Compromised}: 22 pairs (52.4\%)
\end{itemize}

The overall detection rate (\textsc{Warning} or \textsc{Compromised}) is \textbf{83.3\%} (35/42 pairs).

\subsubsection{Analysis by Manipulation Type}

\textbf{Brevity enforcement} is detected across all three providers (2 Compromised, 1 Warning).
The dominant deviated features are \texttt{length\_words} ($z = 8.1$--$18.0$) and \texttt{length\_chars} ($z = 6.5$--$13.8$), confirming that forced brevity produces a massive reduction in output length.
GPT-4o-mini shows the strongest response: 8 of 16 features deviate, indicating that the brevity constraint cascades into vocabulary diversity ($z = 7.2$), sentence structure, and formatting.

Claude Haiku produces only a \textsc{Warning} ($w_z = 8.1$, 3 deviated) rather than \textsc{Compromised}, because Claude's normal responses are already relatively concise (3.44 sentences on average), so the brevity constraint produces a smaller \emph{relative} shift.
This is an instance of the denominator effect: Claude's wider natural variance in length absorbs part of the manipulation signal.

\textbf{Uncertainty injection (hedging)} is detected across all three providers (3 Compromised).
The targeted feature \texttt{hedge\_ratio} is the top deviation in all three models, with $z = 6.2$ (GPT-4o-mini), $12.5$ (Claude Haiku), and $8.5$ (Grok-3).
Claude Haiku shows the highest z-score for hedging---ironically, the model that normally hedges least is most affected when forced to hedge.

This result validates the security weight design: \texttt{hedge\_ratio} carries weight $w = 1.5$, amplifying its contribution to the weighted z-score.
Without the weight, the raw z-scores would still exceed $\sigma = 3.0$, but the margin would be smaller.

\textbf{Format coercion} shows provider-dependent detection.
GPT-4o-mini obeys the format constraint strictly, producing dramatically shorter, structured outputs (\textsc{Compromised}, $w_z = 14.5$).
Claude Haiku partially complies (\textsc{Warning}, $w_z = 3.8$).
Grok-3 essentially ignores the constraint and generates normal-looking responses (\textsc{Clean}, $w_z = 1.8$).

This last case is instructive: the tampering is \emph{undetectable} precisely because it is \emph{ineffective}.
The system prompt manipulation fails to change Grok-3's behavior, so there is nothing to detect.
This is a correct negative, not a false negative.

\subsection{Experiment 5: Temperature Drift}
\label{sec:exp5}

Both temperature extremes tested on GPT-4o-mini produce \textsc{Clean} results:
\begin{itemize}[nosep]
    \item $T = 0.2$ vs.\ $T = 0.7$: $w_z = 1.17$, 0 deviated features
    \item $T = 1.2$ vs.\ $T = 0.7$: $w_z = 0.89$, 0 deviated features
\end{itemize}

\begin{figure}[H]
\centering
\includegraphics[width=\columnwidth]{figures/fig5_temperature_drift.png}
\caption{Feature-level z-scores for temperature drift. No feature exceeds the $3\sigma$ threshold at either temperature extreme. Temperature is a ``stealth'' parameter invisible to signal-based monitoring.}
\label{fig:temp}

Temperature changes within the commonly used range ($T \in [0.2, 1.2]$) do not produce statistically detectable behavioral shifts.
Even the maximum observed z-score ($w_z = 1.17$ at $T = 0.2$) is far below the $3\sigma$ threshold, and the lowest temperature actually produces a \emph{higher} z-score than the highest temperature ($1.17$ vs.\ $0.89$), suggesting the effect is noise rather than signal.

This establishes a clear \textbf{detection boundary}: temperature is a ``stealth'' parameter that can be manipulated without triggering signal-based monitoring.
This is both a limitation of the approach and a positive result---it means normal temperature tuning by providers (within standard ranges) will not cause false alarms.

\section{Discussion}
\label{sec:discussion}
%% ═══════════════════════════════════════════════════════════════

\subsection{Threat Model Boundaries}

Our experiments establish a layered detection capability:

\textbf{Tier 1 (Reliable detection):}
Cross-provider model swaps involving Claude ($w_z > 10$ in all cases), system prompt tampering via brevity or hedging manipulation ($w_z > 6$), and architectural-generation gaps within the same provider (GPT-3.5 vs.\ GPT-4 family, $w_z > 5.9$).

\textbf{Tier 2 (Partial detection):}
Intra-family model swaps produce \textsc{Warning} in most cases but rarely \textsc{Compromised}.
Format coercion is detected when the model complies strongly (GPT-4o-mini) but not when compliance is weak (Grok-3).

\textbf{Tier 3 (Undetectable):}
Same-generation model swaps (Grok-3 $\leftrightarrow$ Grok-3-mini), cross-provider behavioral twins (GPT-4o-mini $\leftrightarrow$ Grok-3), and temperature manipulation ($T \in [0.2, 1.2]$).

\subsection{The Behavioral Convergence Hypothesis}

The most unexpected finding is that GPT-4o-mini and Grok-3, developed independently by OpenAI and xAI respectively, produce statistically indistinguishable behavioral profiles.
Multiple explanations are possible:

\begin{enumerate}[nosep]
    \item \textbf{RLHF convergence:} Both models undergo reinforcement learning from human feedback, and human preferences for ``balanced analytical responses'' may converge to a narrow behavioral attractor regardless of base architecture.
    \item \textbf{Training data overlap:} Both models are trained on web-scale corpora that likely share significant overlap, encoding similar conventions for response structure.
    \item \textbf{Feature limitations:} Our 16 surface-level features may not capture the dimensions along which these models truly differ (e.g., reasoning depth, factual accuracy, creative originality).
\end{enumerate}

This convergence has practical implications: a provider could swap between these two models without detection.
Whether this constitutes a real threat depends on whether the consumer cares about model identity per se or about model \emph{quality}---if both models deliver equivalent quality, the swap may be economically motivated but not harmful.

\subsection{Claude's Distinctiveness}

Claude models occupy a unique position in the behavioral space.
The structural differences are not subtle: Claude Haiku writes 3.44 sentences per response where GPT-4o-mini writes 22.40; Claude's average sentence length is 103.29 words vs.\ 13.38 for GPT-4o-mini.
These are not marginal deviations---they represent fundamentally different response architectures (flowing paragraphs vs.\ bulleted lists).

This suggests that Anthropic's training and alignment process encodes a distinctive ``voice'' that is qualitatively different from the OpenAI/xAI convergence point.
For monitoring purposes, this is advantageous: any swap involving Claude will be immediately detected.

\subsection{The Gemini Lesson}

The Gemini~2.0~Flash results, while preliminary, illustrate an important methodological point.
When rate-limit errors corrupt a baseline, the near-zero standard deviations produce astronomically inflated z-scores ($w_z > 10^{12}$).
This is not a bug---it is the monitoring system correctly identifying that the ``baseline'' behavior (error messages) is incompatible with any real model's behavior.

In production, this pattern could serve as an automated indicator of API health: a baseline whose standard deviations approach zero is likely corrupted and should trigger a re-calibration alert rather than false compromise detection.

\subsection{Practical Recommendations}

Based on our findings, we recommend the following monitoring strategy for API consumers:

\begin{enumerate}[nosep]
    \item \textbf{Establish baselines during evaluation:} Before committing to a model, run 25--50 controlled prompts to capture the behavioral profile. Store the \texttt{SecurityBaseline} JSON.
    \item \textbf{Monitor continuously:} Run LLM~EKG (or equivalent) on a sample of production requests. Check against the stored baseline at $\sigma = 3.0$.
    \item \textbf{Baseline on the tightest model:} When monitoring multiple models, use the one with lowest natural variance as the primary baseline for maximum sensitivity.
    \item \textbf{Watch \texttt{length\_words}:} This single feature is the most reliable indicator of behavioral manipulation across all conditions tested.
    \item \textbf{Combine with task-specific checks:} Signal-based monitoring detects \emph{that} behavior changed, not \emph{why}. Complement with domain-specific quality metrics for full coverage.
\end{enumerate}

\subsection{Limitations and Threats to Validity}

\begin{enumerate}[nosep]
    \item \textbf{Prompt homogeneity:} All experiments use a single prompt format (analytical comparison). Results may differ for creative writing, code generation, or conversational prompts.
    \item \textbf{Sample size:} 25 responses per condition provides adequate statistical power for large effects ($z > 3$) but may miss subtle shifts. Larger samples would reduce baseline variance.
    \item \textbf{Gemini data quality:} Free-tier rate limits corrupted the Gemini baseline. A paid-tier replication is needed for definitive Gemini results.
    \item \textbf{Language dependency:} The hedging and assertion lexicons cover English and Spanish. Extension to other languages requires adding word lists.
    \item \textbf{Temporal stability:} All data was collected in a single session (March 2026). Model behavior may drift over time as providers update weights or infrastructure.
    \item \textbf{Combined attacks:} We test each threat scenario in isolation. A sophisticated attacker could combine multiple subtle manipulations (e.g., slight brevity + mild temperature change) that individually fall below $3\sigma$ but jointly alter behavior.
    \item \textbf{Feature completeness:} 16 surface-level features cannot capture all behavioral dimensions. Semantic quality, reasoning accuracy, and factual correctness are beyond scope.
\end{enumerate}

\section{Ethical Considerations}
\label{sec:ethics}
%% ═══════════════════════════════════════════════════════════════

This work is motivated by the \emph{consumer's right to verify} what they are paying for.
We believe API consumers deserve transparency about the model serving their requests, just as consumers deserve accurate product labels in any other market.

We acknowledge the dual-use potential: the same fingerprinting techniques that protect consumers could theoretically be used by attackers to craft evasion strategies.
We mitigate this risk by publishing the methodology openly---security through obscurity is not a viable long-term strategy, and defenders benefit more from open tools than attackers do.

All API calls in this study were made through properly authenticated accounts within the terms of service of each provider.
No jailbreaking, prompt injection, or adversarial techniques were used.
The prompts are benign analytical comparisons that any normal user might submit.

\section{Conclusion}
\label{sec:conclusion}
%% ═══════════════════════════════════════════════════════════════

We demonstrate that statistical monitoring of LLM outputs, using 16 domain-agnostic features and weighted z-score comparison, provides a practical defense against behavioral manipulation of API-served language models.

The system reliably detects:
\begin{itemize}[nosep]
    \item Cross-provider model swaps: 83.3\% of directed pairs (35/42).
    \item System prompt tampering: 77.8\% of conditions (7/9), with z-scores 3.8--18.0.
    \item Architectural-generation gaps: GPT-3.5 vs.\ GPT-4 family within OpenAI.
\end{itemize}

The system does not detect:
\begin{itemize}[nosep]
    \item Same-generation intra-family swaps (Grok-3 $\leftrightarrow$ Grok-3-mini).
    \item Cross-provider behavioral twins (GPT-4o-mini $\leftrightarrow$ Grok-3).
    \item Temperature manipulation within $T \in [0.2, 1.2]$.
\end{itemize}

These boundaries are not failures---they define the principled scope of what surface-level statistical analysis can and cannot achieve.
For production deployment, we recommend combining signal-based monitoring with task-specific quality checks for comprehensive coverage.

The discovery that independently developed models can be statistically indistinguishable raises broader questions about behavioral convergence in the LLM industry.
As models are trained on overlapping data and optimized for similar human preferences, the behavioral diversity that enables fingerprinting-based detection may erode over time.
Monitoring the evolution of these fingerprints could itself become a valuable research direction.
